Windows下Nginx的启动、停止等命令

  • 1、启动：
    C:\server\nginx-1.0.2>start nginx或
    C:\server\nginx-1.0.2>nginx.exe

  • 2、停止：
    C:\server\nginx-1.0.2>nginx.exe -s stop或
    C:\server\nginx-1.0.2>nginx.exe -s quit
    注：stop是快速停止nginx，可能并不保存相关信 息；quit是完整有序的停止nginx，并保存相关信息。

  • 3、重新载入Nginx：
    C:\server\nginx-1.0.2>nginx.exe -s reload
    当配置信息修改，需要重新载入这些配置时使用此命令。

  • 4、重新打开日志文件：
    C:\server\nginx-1.0.2>nginx.exe -s reopen

  • 5、查看Nginx版本：
    C:\server\nginx-1.0.2>nginx -v

windows环境下 elk 收集监控nginx日志

1.前言

对于互联网公司来说，nginx的请求日志简直就是一座金矿，如果不能充分利用，简直太可惜了。
初期一般都是输出到日志文件，要查什么就awk\grep\uniq\sort…，能满足不少统计需求，但最大的缺点是不直观，不方便监控（目前虽然用了ELK，但是有些信息还是用shell统计，两者互补）。
整理下实施ELK最起码要实现的需求：

  • 查询条件（精确匹配）：一级域名、二级域名、客户真实IP、HTTP状态码、HTTP方法、request_time、response_time、代理IP、body_bytes_sent
  • 查询条件（模糊匹配）：url（如查找SQL注入关键字）、refere（流量来源）、agent（如查找搜索引擎）、近期（1周、1个月）内整体请求量走势情况；如果发现总体走势异常，要很方便找到那个域名走势异常；
  • 过去一个周期内（1天、1周、1月）所有请求构成，按不同域名出饼图；
  • 实时监控爬虫IP过高的频率访问（如单个IP1分钟请求超过100次报警）；
  • 实时监控500状态请求情况（如单个域名1分钟出现30个500就报警）；

3.nginx配置

  • 下载nginx
  • 解压
  • 进入config目录，修改nginx.conf

nginx.conf
日志以json格式输出，方便logstash解析；

  • 启动 nginx (有很多种方法启动 nginx ）

    • (1)直接双击nginx.exe，双击后一个黑色的弹窗一闪而过
    • (2)打开cmd命令窗口，切换到nginx解压目录下，输入命令 nginx.exe 或者 start nginx ，回车即可

  • 检查nginx是否启动成功
    直接在浏览器地址栏输入网址 http://localhost:80/，回车，出现以下页面说明启动成功

  • 关闭nginx

    如果使用cmd命令窗口启动nginx，关闭cmd窗口是不能结束nginx进程的，可使用两种方法关闭nginx
    • (1)输入nginx命令 nginx -s stop(快速停止nginx) 或 nginx -s quit(完整有序的停止nginx)
    • (2)使用taskkill taskkill /f /t /im nginx.exe

  • 参考博客：https://www.cnblogs.com/jiangwangxiang/p/8481661.html/

logstash是作为数据采集，功能和flume类似

  • 进入bin目录新建配置文件logstash.conf
  • 命令行进入logstash 目录 ，启动logstash bin\logstash -f logstash.conf
  • 浏览器访问:http://localhost:9600/

如何在多台电脑上维护hexo

使用hexo搭建的博客，以前电脑中创建过hexo根目录，
然后中途更换过电脑，以前电脑上的配置文件 在现电脑上都没有了
解决方法：
如果想要继续维护以前的博客，那么：

首先，我们要把以前github上博客的仓库clone下来，通过实践，我们知道，这个clone下来的文件夹就是以前本地博客根目录下的public文件夹
我们可以自己本地安装hexo之后，自己创建一个public文件夹，然后把clone下来的文件全部放进去

为了以后不再这么麻烦，我们可以直接在GitHub上备份一下我们的hexo配置文件，以后再遇到这种情况的时候，就可以直接clone博客配置文件到本地， clone博客内容文件夹到本地，然后把内容文件夹该名称public放在配置文件夹根目录即可

linux flume监控文件输出到kafka

flume监控文件

  • flume的安装与测试

    • 下载flume（windows与linux相同）
    • 解压：tar xvzf apache-flume-1.8.0-bin.tar.gz
    • 进入conf目录：cd flume/conf
    • 修改或创建配置文件：kafka.conf

  • 启动flume: bin/flume-ng agent --conf conf --conf-file /root/apache-flume-1.8.0-bin/conf/kafka.conf --name agent -Dflume.root.logger=INFO,console

kafka

  • 安装zookeeper

    • 下载zookeeper
    • 解压 tar -zxvf zookeeper
    • zookeeper目录下创建data和log目录:mkdir data mkdir log
    • 进入config目录将zoo_sample.cfg修改为zoo.cfg：mv zoo_sample.cfg zoo.cfg
    • 编辑zoo.cfg，将dataDir修改为创建的data路径dataDir=/root/zookeeper-3.4.13/data
    • 启动zk： bin/zkServer.sh start conf/zoo.cfg
    • 查看服务启动状态：bin/zkServer.sh status
    • 或者通过jps查看zk的线程

  • 安装kafka

    • 下载kafka
    • 解压 tar -zxvf kafka
    • kafka目录下创建kafka-logs目录 mkdir kafka-logs
    • 进入config目录修改配置文件cd kafaka/config vi server.properties
      log.dirs=/root/kafka_2.11-2.1.1/kafka-logs
    • 启动kafka server： bin/kafka-server-start.sh config/server.properties
    • 创建topic（flume配置文件中的topic与这个topic相同）：bin/kafka-topics.sh --create --zookeeper localhost:2181 --replication-factor 1 --partitions 1 --topic qstopic
    • 启动（新版）kafka 消费者：bin/kafka-console-consumer.sh --bootstrap-server localhost:9092 --topic qstopic --from-beginning
    • 进入flume监控目录，修改.log目录文件内容 echo '111' >> test.log

error

  • 启动kafka的时候出现的错误，此时检查zk是否正常启动
  • 安装zookeeper时候，可以查看进程启动，但是状态显示报错：Error contacting service. It is probably not running
    • 查看配置文件发现没错误，将之前启动的zk线程kill掉，重新启动
